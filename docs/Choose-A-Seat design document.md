# Choose-A-Seat design document

1. 这是一个微信小程序 所以需要对微信进行适配 以及告诉我怎么进行部署
2. 这个小程序的主要功能有
   1. 首先需要用户管理系统 有两种用户: 1. 管理员 2. 普通用户
   2. 管理员可以创建一个项目(生成一个key) 可以设置密码 一个项目里面可以新增车辆 每台车可以命名 可以选择车子的可用座位数目(座位排列按照市面上基本的排列格式进行排列) 
   3. 普通用户可以通过key找到对应项目 输入密码进入 可以选择其中一个车辆中的一个座位(如果一个用户选择了该座位其他人不能选择: 参考飞机选座)
   4. 管理员可以对已经选择座位的用户座位进行修改, 对应普通用户会收到提醒
3. 告诉我微信小程序平台能否提供数据库 如果不行我会提供给你一个数据库

-----



### 1. 项目规模和使用场景

这会决定用 **CloudBase Serverless** 还是 **自建后端 + 自建数据库**。

我需要知道：

1. 这个小程序是给谁用？
    旅行团、大巴选座。

2. 预计同时在线人数是多少？

   10～100 人：CloudBase 很适合。

3. 一个项目一般有多少辆车？
    不超过十辆。

4. 每辆车最多多少座？
   最好包含市面上常见的座位数就行

5. 项目是一次性的，还是长期复用？
    需要历史记录,记录每个人选的座位,可以根据历史记录创建新的项目(只有对应车辆没有座位被选择)。

------



### 2. 用户身份和登录方式

这会影响数据库设计、权限设计和微信审核。

我需要知道：

1. 普通用户是否需要微信登录？
   **必须使用微信 openid 作为唯一用户标识**，否则无法可靠防止同一个人重复选座。
2. 普通用户是否需要填写姓名、手机号、学号、工号？
   - 只需要微信名就行
   - 备注可选
3. 管理员怎么产生？
   - 管理员账号需要后台审核。
   - 后期可能管理员需要付费
4. 一个项目能不能有多个管理员？
    暂时不需要。

-----



### 3. 选座规则

这是 spec 里最重要的业务逻辑。

我需要知道：

1. 一个用户只能选一个座位吗？
    一个用户只能选一个座位

2. 用户选完后能不能自己改座？

   - 可以取消座位。
   - 在截止时间之前
   - 改座需要重新抢占。

3. 是否需要选座截止时间？
    可以由管理员自行设置

4. 是否需要锁座倒计时？
   不需要，点击确认后直接占用即可，MVP 更简单。

5. 管理员改用户座位时，如果目标座位已经有人怎么办？
    需要确定规则：

   - 允许管理员手动重新分配。

6. 是否有特殊座位？

   可以设置不可选座 司机位默认不可选择

------



### 4. 车辆和座位布局

这会决定前端组件和数据库 schema。

我需要知道：

1. 你需要支持哪些车？ 市面上大部分车型都支持
   - 5 座小车
   - 7 座商务车
   - 9 座车
   - 33 座中巴
   - 45 / 49 / 53 座大巴
2. 座位布局是固定模板，还是管理员可以自定义？
   - MVP：内置常见模板，管理员只选择模板。
3. 是否需要显示“司机位”“车门”“过道”？
    只需要显示司机位
4. 座位编号规则是什么？
   - 左 1、左 2、右 1、右 2

------

### 5. 通知机制

你提到管理员改座后普通用户会收到提醒。这个可以做，但需要注意：微信小程序通常使用**订阅消息**，用户需要先授权订阅，不能随便无限制推送。第三方资料和微信生态实践都强调订阅消息需要用户主动订阅/授权后才能下发。

我需要知道：

1. 你希望怎么通知用户？

   微信服务通知 / 订阅消息

2. 是否必须“实时通知”？
   无需实时通知

3. 管理员修改座位时，用户是否需要确认？
    例如：

   - 管理员直接改，用户只收到通知。

------

### 6. 管理员后台形态

我需要知道管理员是在小程序里操作，还是需要单独 Web 后台。

**全部在微信小程序里完成**
 适合 MVP，开发量小，部署简单。
